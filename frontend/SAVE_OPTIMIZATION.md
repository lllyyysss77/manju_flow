# 剧本创作模块保存优化说明

## 已实现的优化

### 1. 自动保存间隔优化 ✅
- **原来**: 5 秒自动保存
- **现在**: 2 秒自动保存
- **位置**: `ScriptEditor.tsx` 行 1347-1363

**说明**: 缩短自动保存间隔可以更快捕获用户的编辑内容，减少数据丢失风险。

---

### 2. 快捷键手动保存 ✅
- **Windows/Linux**: `Ctrl+S`
- **Mac**: `Cmd+S`
- **位置**: `ScriptEditor.tsx` 行 1365-1389

**功能**:
- 阻止浏览器默认的保存对话框
- 智能识别当前编辑内容（场景或章节梗概）
- 保存成功后显示 Toast 提示
- 无改动时提示"无改动，无需保存"

---

### 3. 本地缓存备份 ✅
**位置**: `useScriptEditorReducer.ts` 行 451-515, 527-570

**工作原理**:
1. **保存前备份**: 每次调用 API 保存前，先将数据备份到 `localStorage`
2. **备份键格式**:
   - 场景: `manju_scene_${sceneId}`
   - 章节梗概: `manju_synopsis_${chapterId}`
3. **保存成功清理**: API 保存成功后，自动删除 localStorage 备份
4. **备份恢复**: 页面加载时检查是否有未同步的备份，提示用户恢复

**恢复逻辑**:
- 只恢复 1 小时内的备份（超过 1 小时自动清除）
- 用户可选择"恢复备份"或"丢弃备份"
- 恢复后需手动保存到服务器

---

### 4. 保存失败自动重试 ✅
**位置**: `useScriptEditorReducer.ts` 行 451-515, 527-570

**重试策略**:
- 最多重试 **3 次**
- 延迟递增：第 1 次等 1 秒，第 2 次等 2 秒，第 3 次等 3 秒
- 重试期间显示进度："正在重试 (1/3)"

**失败处理**:
- 3 次重试全部失败后，显示错误提示
- 数据已备份到 localStorage，不会丢失
- 用户可稍后手动重试保存

---

### 5. 更明显的保存状态提示 ✅
**位置**: `ScriptEditor.tsx` 行 1765-1830

**视觉反馈**:

#### 状态指示点
- 🔴 **红色闪烁**: 保存失败
- 🟠 **橙色闪烁**: 有未保存更改
- 🟡 **黄色动画**: 保存中/重试中（带 ping 动画）
- 🟢 **绿色**: 已保存

#### 状态文字
- "保存失败" + 错误详情
- "未保存" (有改动时)
- "保存中..."
- "正在重试 (1/3)"
- "已保存" + 上次保存时间

#### 未保存徽章
- 有改动时显示橙色动画徽章："有未保存更改"

#### 手动保存按钮
- 显示快捷键提示: "(Ctrl+S)"
- 保存中禁用，防止重复点击

---

### 6. 离开页面前提醒 ✅
**位置**: `ScriptEditor.tsx` 行 1391-1403

**功能**:
- 有未保存更改时关闭/刷新页面会弹出确认对话框
- 防止用户意外丢失编辑内容

## 使用场景示例

### 场景 1: 正常编辑和保存
1. 用户编辑场景内容
2. 状态点变为 🟠 橙色，显示"未保存"徽章
3. 2 秒后自动保存
4. 状态点变为 🟢 绿色，显示"已保存"

### 场景 2: 网络断线
1. 用户编辑场景内容
2. 2 秒后触发自动保存
3. API 请求失败（网络断线）
4. 数据已备份到 localStorage
5. 系统自动重试 3 次（延迟 1s, 2s, 3s）
6. 如果 3 次都失败，显示错误提示
7. 用户恢复网络后，可按 `Ctrl+S` 手动重试

### 场景 3: 浏览器崩溃
1. 用户编辑场景内容
2. 触发自动保存，数据备份到 localStorage
3. 浏览器突然崩溃
4. 用户重新打开页面
5. 检测到本地备份，弹出恢复提示
6. 用户点击"恢复备份"，内容恢复
7. 用户手动保存到服务器

### 场景 4: 快捷键保存
1. 用户正在编辑
2. 按 `Ctrl+S` 或 `Cmd+S`
3. 立即保存当前内容
4. 显示 Toast: "场景已保存 (Ctrl+S)"

## 技术细节

### localStorage 存储格式

**场景备份**:
```json
{
  "id": 123,
  "chapterId": 456,
  "description": "场景描述",
  "dialogue": "台词",
  "cameraMovement": "运镜",
  "transitionEffect": "转场",
  "status": "DRAFT",
  "backupTime": "2024-02-10T12:34:56.789Z"
}
```

**章节梗概备份**:
```json
{
  "synopsis": "章节梗概内容",
  "backupTime": "2024-02-10T12:34:56.789Z"
}
```

### 重试机制实现
```typescript
const persistScene = async (chapterId, scene, retryCount = 0) => {
  // 1. 备份到 localStorage
  localStorage.setItem(`manju_scene_${scene.id}`, JSON.stringify(scene));

  try {
    // 2. 调用 API 保存
    await sceneApi.update(...);

    // 3. 保存成功，清除备份
    localStorage.removeItem(`manju_scene_${scene.id}`);
  } catch (err) {
    // 4. 保存失败，自动重试
    if (retryCount < 3) {
      await new Promise(resolve => setTimeout(resolve, (retryCount + 1) * 1000));
      return persistScene(chapterId, scene, retryCount + 1);
    }
    // 5. 重试失败，提示错误
    throw new Error('保存失败，已重试3次');
  }
};
```

## 性能考虑

1. **localStorage 大小限制**:
   - 现代浏览器通常支持 5-10MB
   - 单个场景数据 < 10KB，足够使用

2. **重试延迟设计**:
   - 递增延迟避免服务器压力
   - 最长等待 6 秒（1+2+3）

3. **自动清理**:
   - 保存成功后立即清除备份
   - 超过 1 小时的备份自动清除
   - 避免 localStorage 膨胀

### 7. **保存队列优化** 🚦
**位置**: `useScriptEditorReducer.ts` 行 291-668

**问题**: 快速切换场景时，可能触发多个并发保存请求，导致数据冲突或顺序错乱

**解决方案**: 引入保存队列机制，确保保存操作按顺序执行

**工作原理**:
1. **队列管理**: 所有保存请求先进入队列，按FIFO顺序执行
2. **任务合并**: 检测队列中相同ID的任务，只保留最新的（去重优化）
3. **并发控制**: 使用 `isProcessingQueueRef` 标志确保同时只有一个任务在执行
4. **状态可见**: 实时更新 `saveQueueSize` 显示队列中的任务数量

**核心数据结构**:
```typescript
interface SaveTask {
  id: string;                    // 唯一标识: "scene_123" 或 "synopsis_456"
  type: 'scene' | 'synopsis';    // 任务类型
  chapterId: number;             // 章节ID
  scene?: Scene;                 // 场景数据（scene类型）
  synopsis?: string;             // 梗概内容（synopsis类型）
  timestamp: number;             // 任务创建时间戳
}
```

**队列处理流程**:
1. 用户编辑触发保存 → `persistScene()` / `persistChapterSynopsis()`
2. 创建 SaveTask 并加入队列（检查重复ID，合并最新）
3. 触发 `processSaveQueue()` 队列处理器
4. 逐个执行任务：调用 `persistSceneInternal()` / `persistChapterSynopsisInternal()`
5. 任务完成后从队列移除，更新 `saveQueueSize`

**任务去重逻辑**:
- 添加任务时：如果队列中已有相同ID的任务，替换为最新的
- 处理任务时：在执行前再次检查后续是否有重复，如有则跳过当前任务

**UI 提示**:
- 队列状态文字：`队列中：2 个任务`（9px，蓝色文字）
- 队列徽章：当队列 > 1 时显示蓝色动画徽章 `队列: 3`（带 ping 动画）

**优化效果**:
- ✅ 避免并发保存导致的数据覆盖
- ✅ 减少重复请求，节省网络带宽
- ✅ 快速切换场景时数据不丢失
- ✅ 队列状态透明，用户可预期等待时间

**使用场景**:
- 用户快速连续编辑多个场景
- 自动保存触发时用户又手动触发保存
- 网络慢导致前一个请求还未完成，又触发新的保存
